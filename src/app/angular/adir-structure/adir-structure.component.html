Angular Directory Structure



Application structure and Angular modules

  • All of the app's code goes in a folder named src. All feature areas are in their own folder, with their own Angular module.
  • All content is one asset per file. Each component, service, and pipe is in its own file.
  • All third party vendor scripts are stored in another folder and not in the src folder. Use the naming conventions for files.
  • Keep a flat folder structure as long as possible.
  • Consider creating sub-folders when a folder reaches seven or more files.
  • Consider configuring the IDE to hide distracting, irrelevant files such as generated .js and .js.map files.
  • Creating a folder for a component when it has multiple accompanying files (.ts, .html, .css and .spec).
  • Create folders named for the feature area they represent.
  • Create an Angular module for each feature area.
  • Create an Angular module in the app's root folder (e.g., in /src/app).
  • Naming the root module app.module.ts.
  • Do not provide services in shared modules. Services are usually singletons that are provided once for the entire application or in a particular feature module.

All configuration files with meaning

  1. Package.json: Contain all packages and its versions that our app’s required. These packages install and maintained with npm.
  2. Systemjs.config: module loader that can import modules at run time in any of the popular formats used today (CommonJS, UMD, AMD, ES6).
  3. tsconfig.json: TypeScript compiler configuration file. It guides the compiler to generate JavaScript files.
  4. module.ts: help organize an application into cohesive blocks of functionality. An Angular Module is a class adorned with the @NgModule decorator function. @NgModule takes a metadata object that tells Angular how to compile and run module code. It identifies the module's own components, directives and pipes, making some of them public so external components can use them.
  5. component.ts: Template(view layout, created with HTML, includes binding & directives) Class (code supporting view, created with typescript, properties, method) Metadata: Extra data for angular, define with decorator(selector:).
  6. main.ts: the file that bootstraps (start) your Angular 2 application.
  7. angular-cli.json: A configuration file for the angular command line interface.
  8. Karma.confi.json: A configuration script for the Karma testing framework. You can use Karma to run your *.spec.ts files on multiple browsers.
  9. protractor.conf.js: A configuration file for the Protractor end-to-end testing framework. End-to-end testing allows you to test your app by mimicking a user interacting with it. You’ll use this with your tests found in the e2e directory.
  10. tslint.json: it’s a static analysis tools for typescript code.
  11. README.md: A place to describe your project.
  12. favicon.ico: The icon that will show in the browser tab.
  13. index.html: The html file that the browser will load. In this file you’ll see the `app-root` tag, which is where your bootstrapped Angular 2 application will load.
  14. polyfills.ts: This loads some code that Angular 2 needs to run on all browsers. Add features which are not support by JavaScript to Angular.
  15. styles.css: You can use this for global styles.
  16. test.ts: A test file for running Karma tests.
  17. typings.d.ts: A configuration file for Typings. Typings is a useful tool for adding javascript libraries to your typescript project.
  18. app: Your actual Angular 2 application source!
  19. assets: A place for you to put static assets such as images, icon, text file, etc.
  20. environments: Environments configuration files used by angular-cli, here we store configuration setting of different environment.
  21. .editorconfig: Editor configuration. While working in team all developer should use same setting.
  22. .gitignore: excluding files from Git repository. Managing and versioning source code.
